  <action>
    Create `src/components/WeeklySchedule.tsx` as a `'use client'` component.

    **Props interface:**
    ```typescript
    interface WeeklyScheduleProps {
      // Current selection: Set of strings like "1-morning", "6-evening"
      // Key format: "{dayOfWeek}-{timeOfDay}" where dayOfWeek 0=Sun..6=Sat
      selection: Set<string>
      onChange: (newSelection: Set<string>) => void
    }
    ```

    **Behaviour:**
    - Render 7 day buttons in a row: Sun Mon Tue Wed Thu Fri Sat (0–6).
    - A day is "active" if any key in `selection` starts with `"{day}-"`.
    - Active day button: amber/gold highlight (e.g. `bg-amber-500 text-gray-950 font-bold`). Inactive: muted (`bg-gray-800 text-gray-400`).
    - Clicking an inactive day toggles it on (adds `"{day}-afternoon"` as default — a sensible starting point). Clicking an active day toggles it off (removes ALL `"{day}-*"` keys from selection).
    - When a day is active, render inline below it (or beneath the row for mobile) three time buttons: Morning | Afternoon | Evening.
    - Each time button is independently toggleable. Active time: `bg-amber-700 text-amber-100`. Inactive time: `bg-gray-700 text-gray-400`.
    - Clicking a time button that is already selected removes just that `"{day}-{time}"` key. If removing it means zero times remain for that day, also deactivate the day (so the day toggle reflects "off").
    - Clicking a time button that is not selected adds `"{day}-{time}"` key.
    - On every change, call `onChange(newSelection)` with the new full Set.

    **Styling (D&D dark theme consistent with Phase 2 `/invite/[token]` page):**
    - Background: `bg-gray-950` or transparent (sits inside the form page)
    - Section heading: `text-sm text-gray-400 uppercase tracking-widest mb-3` — "Weekly Schedule"
    - Day buttons: `rounded-md px-3 py-2 text-sm font-medium transition-colors`
    - Use `flex flex-wrap gap-2` for day button row so it wraps on narrow screens
    - Time sub-options: `flex gap-2 mt-2 ml-1` with smaller text (`text-xs`)

    **No external dependencies** — pure Tailwind + React state.

<task type="auto">
  <name>Task 2: AvailabilityCalendar component</name>
  <files>src/components/AvailabilityCalendar.tsx</files>
  <action>
    Create `src/components/AvailabilityCalendar.tsx` as a `'use client'` component.

    **Props interface:**
    ```typescript
    interface AvailabilityCalendarProps {
      planningWindowStart: string   // ISO string from server, e.g. "2026-03-01T00:00:00.000Z"
      planningWindowEnd: string     // ISO string from server
      // weeklySelection: Set<string> — same format as WeeklySchedule ("1-morning" etc)
      weeklySelection: Set<string>
      // overrides: map of ISO date string "YYYY-MM-DD" -> "free" | "busy"
      overrides: Map<string, 'free' | 'busy'>
      onDateClick: (dateKey: string) => void  // dateKey = "YYYY-MM-DD"
    }
    ```

    **Calendar grid generation (no library):**
    Build a pure function `buildMonthGrid(year: number, month: number): (Date | null)[][]` that:
    1. Gets the first day of the month and its day-of-week (0=Sun)
    2. Generates an array of: `startDow` null padding cells + Date objects for each day of month
    3. Pads the end to a multiple of 7
    4. Splits into rows of 7

    **Rendering:**
    - Show each month in the planning window from `windowStart` to `windowEnd` (inclusive).
    - Derive the range of months to show: from `(windowStart.getUTCFullYear(), windowStart.getUTCMonth())` to `(windowEnd.getUTCFullYear(), windowEnd.getUTCMonth())`.
    - For each month: render a month header (`"March 2026"` style with `toLocaleDateString('en-GB', { month: 'long', year: 'numeric', timeZone: 'UTC' })`), then the day-of-week header row (S M T W T F S), then the grid.
    - Multiple months stacked vertically (scrollable — per Claude's discretion decision).
    - Day-of-week header: `text-xs text-gray-500 text-center`

    **Date cell visual states:**
    Derive each cell's state using this logic:
    1. If date is `null` (padding): empty cell
    2. If date is outside planning window: render greyed out, not clickable (`text-gray-700 cursor-default`)
    3. Check overrides map: if date key exists → override state
    4. Check weekly pattern: derive dayOfWeek from date; if any `"{dow}-*"` key in weeklySelection → pattern-free
    5. Otherwise: pattern-busy

    **Visual styles:**
    - `pattern-free` (tinted from weekly pattern): `bg-amber-900/40 text-amber-200` with no marker — subtle amber tint
    - `pattern-busy` (no availability): `text-gray-500`
    - `override-free` (explicitly marked free): `bg-green-800 text-green-200 ring-1 ring-green-500` — distinct green marker
    - `override-busy` (explicitly marked busy): `bg-red-900 text-red-300 ring-1 ring-red-700` — distinct red marker
    - All in-window cells are `cursor-pointer rounded-md text-sm text-center py-1.5 transition-colors`

    **Click behaviour:**
    - Dates outside the window: not clickable (no onClick handler)
    - In-window dates: call `onDateClick(dateKey)` where `dateKey` is `"YYYY-MM-DD"` formatted from the UTC date:
      ```typescript
      const dateKey = `${date.getUTCFullYear()}-${String(date.getUTCMonth() + 1).padStart(2, '0')}-${String(date.getUTCDate()).padStart(2, '0')}`
      ```
    - The parent AvailabilityForm handles the toggle logic (cycle: no override → busy override → free override → no override, as per CONTEXT.md: "Clicking an already-overridden date again reverts it to the weekly-pattern state").

    **Date parsing (UTC-safe):**
    - Parse `planningWindowStart` and `planningWindowEnd` with `new Date(isoString)` — these are full ISO strings.
    - Compare dates using UTC methods (`getUTCFullYear`, `getUTCMonth`, `getUTCDate`) to avoid timezone shift.

    **No external library** — custom grid only (research explicitly states: do not use react-day-picker or similar).
